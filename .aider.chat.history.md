# PieLauncher - Architecture Summary

## Overview
PieLauncher is an Android launcher application built with **Jetpack Compose** and **Kotlin**. It provides a pie/fan-style menu interface for quick app launching, designed for one-handed operation with support for both right-hand and left-hand layouts.

## Core Architecture

### **UI Layer (Jetpack Compose)**
- **PieOverlay.kt** - Main composable that orchestrates the entire UI
- **FanMenu.kt** - Renders the pie/fan menu with app icons
- **CenterButton.kt** - Central touch trigger area
- **AppPickerPanel.kt** - UI for selecting/configuring apps
- **AppList.kt** - Displays available apps for selection

### **State Management**
- **PieController.kt** - Main controller managing gesture input and state transitions
  - Handles finger down/move/up events
  - Determines selected slice based on finger position
  - Manages dead zone and activation radius logic
- **PieState.kt** - Immutable state data class (center position, selected slice, edit mode)
- **LauncherMode.kt** - Enum for NORMAL vs CONFIGURATION modes

### **Configuration & Persistence**
- **PieConfiguration.kt** - Data model for launcher configuration with multiple profiles
- **Profile.kt** - Represents a single profile with 6 app slots
- **ConfigurationStore.kt** - SharedPreferences-based persistence layer
  - Saves/loads configurations per profile and slot
  - Supports 3 profiles with 6 slots each

### **App Management**
- **AppRegistry.kt** - Queries installed apps via PackageManager
  - Lists all launchable apps
  - Finds specific apps by package name
- **AppInfo.kt** - Data class representing an app (label, package, activity)
- **IconLoader.kt** - Loads and caches app icons as ImageBitmaps
- **Launcher.kt** - Launches apps via Intent with ComponentName

### **Layout & Geometry**
- **FanSlots.kt** - Calculates icon positions in the fan layout
  - `position()` - Computes icon coordinates based on slot and layout
  - `closest()` - Determines which slot is nearest to finger position
- **LauncherLayout.kt** - Enum for RIGHT_HAND vs LEFT_HAND layouts
- **PieConstants.kt** - UI constants (radius, icon size, dead zone, etc.)

### **Utilities**
- **WallpaperLauncher.kt** - Opens system wallpaper picker

### **Entry Point**
- **MainActivity.kt** - ComponentActivity that sets up the Compose UI with PieOverlay

## Key Design Patterns

1. **Object Singletons** - Most utility classes (AppRegistry, IconLoader, Launcher, etc.) are Kotlin objects for global access
2. **Immutable State** - PieState is immutable; PieController uses `mutableStateOf` for reactive updates
3. **Composition over Inheritance** - Heavy use of Compose functions rather than view hierarchies
4. **Data Classes** - Clean data models (AppInfo, Profile, PieConfiguration)
5. **Separation of Concerns** - Clear boundaries between UI, state, persistence, and business logic

## User Interaction Flow

1. User touches screen → `PieController.fingerDown()` activates menu at touch point
2. User drags finger → `PieController.fingerMove()` updates selected slice
3. User releases → App launches if valid slice selected, or enters config mode if in center
4. Configuration mode allows adding/removing apps from slots
5. Multiple profiles (3) can be configured with 6 app slots each

## Technology Stack
- **Language**: Kotlin
- **UI Framework**: Jetpack Compose
- **Build System**: Gradle (Kotlin DSL)
- **Min SDK**: Based on AndroidManifest (not shown in summaries)
- **Architecture**: MVVM-like with reactive state management
